三、为什么 token 会烧得多

这部分是你最关心的。

先说最重要的一句话：

不是所有 token 都花在“修社区产品 bug”本身。
很大一部分 token 花在了“搭控制器 + 修控制器 + 让控制器能继续自动推进”上。

具体分成 5 类：

（一）旧 control-plane 体系反复 loop

根据本地 git 记录，2026-04-03 下午有大量这种提交：
- Update server report: worker loop ...
- Update server report: autopilot loop ...

这些循环说明：
- worker 在持续轮询
- 本地/服务器在多轮读状态、判断、更新文档、写回结果

这类工作本质上非常耗模型：
- 它不是直接修代码
- 但每轮都可能调用一次或多次 Codex
- 对你来说“看不见明显产出”，但账会一直涨


（二）direct controller 本身搭建时发生了多轮试错

主要踩过这些坑：

1. prompt 没有真正喂进 `codex exec`
2. 本机 Windows `cwd` 被错误派发给服务器
3. stale running task 卡住执行器
4. phase 推进 bug，导致 phase 1 结果错误影响 phase 2
5. 本地 controller 401 / token / auth 问题
6. server executor 从社区仓切到独立仓时，一度“服务看起来 running，但 healthz 连不上”

这些都不是用户功能本身，但每个都需要：
- 下任务
- 执行
- 看日志
- 再修

所以 token 会烧得很快。


（三）真实 live 验证比单纯写测试更贵

这轮不是只做单元测试，而是做了很多真实服务器验证：

- 33 -> neko
- neko -> 33
- xhs -> neko
- 33 -> xhs
- xhs -> 33
- live group task flow

每次 live 验证通常不是“1 条命令”：
- 要先下任务给服务器 Codex
- 服务器读代码/日志
- 执行 CLI / systemctl / curl / journalctl / jq / rg
- 再组织结果回传

这就意味着：
- 每一个验证任务本身就是一次完整 agentic 编码/排障任务
- 不是轻量查询


（四）同一个问题被多轮收窄

举几个最典型的例子：

1. `neko` 不回复
- 先怀疑 runtime / targeted
- 再发现是 stale identity
- 再发现 webhook 502
- 再发现 name-targeting 和 live id 有歧义

2. live 线程 bounce / relay
- 一开始像 reply 策略问题
- 后来定位到重复 webhook intake
- 再补 dedupe
- 再重跑 live flow

3. `xhs` onboarding
- attach 成功
- 但要补做 no-regression 和 reattach-stability

这种“收窄 blocker”的过程本身就是 token 黑洞。
你后面要求加入“最多 3 次收窄、最多 3 次修点、5 次修复封顶”，就是因为这部分确实太烧。


（五）控制器和社区工程同时推进，造成额外叠加

当时不是纯社区工程，也不是纯控制器工程，而是两条线并行：

1. 修社区功能
2. 让控制器能持续自动推进社区功能

这导致：
- 每一个社区问题
- 往往还伴随着一次控制器侧的观察、调度、修正

从 token 角度看，就是“双重消耗”。
