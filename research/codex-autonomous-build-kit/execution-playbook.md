# Autonomous Build Execution Playbook

这份文档不是论文复述，而是把论文里的可执行部分揉成一套给 Codex 或多线程执行系统参考的工作流。

## 总目标

输入只有两类：

- 设计文档
- 可运行环境

系统需要尽量少依赖人工介入，完成：

1. 需求归一化
2. 任务拆解
3. 项目推进与线程编排
4. 代码实现
5. 代码审阅
6. 自动测试
7. 真实运行验证
8. 最终交付

## 论文提炼出的 8 条硬规则

1. 不允许一上来直接写代码，必须先做需求归一化。
来源：`ArchCode`

2. 不允许把任务拆成一堆口号式子任务，子任务必须满足“可解、完整、不重复”。
来源：`Agent-Oriented Planning in Multi-Agent Systems`

3. 不允许让所有线程都兼任所有角色，至少要分离需求、实现、审阅、测试。
来源：`MetaGPT`, `TDFlow`, `CodeAgent`

4. 不允许线程长期各干各的而不校准，必须设置对齐检查点。
来源：`RTADev`

5. 不允许只靠大上下文瞎改仓库，必须提供稳定的文件、搜索、测试、日志工具接口。
来源：`SWE-agent`, `SWE-Dev`

6. 不允许“实现线程自己给自己放行”，审阅线程和测试线程必须独立。
来源：`CodeAgent`, `TDFlow`

7. 不允许把“测试通过”当成“真实可用”，必须区分单元测试、集成测试、运行验证。
来源：`TDFlow`, `OSWorld`, `DevEval`

8. 不允许在环境跑不通时继续编故事，环境失败必须进入故障分流。
来源：`SWE-agent`, `OSWorld`

## 推荐线程角色

### 1. Orchestrator
职责：

- 接收设计文档和环境说明
- 维护总状态机
- 分派任务
- 管理线程边界
- 决定是否进入下一阶段

### 2. Requirement Thread
职责：

- 解析设计文档
- 输出需求包
- 标记不明确点
- 补出非功能要求
- 生成验收条件草案

交付物：

- `requirements.md`
- `assumptions.md`
- `acceptance_criteria.md`

### 3. Planning Thread
职责：

- 把需求包拆成任务图
- 建依赖关系
- 指定每个线程的写入边界
- 给出里程碑和回滚点

交付物：

- `task_graph.json`
- `milestones.md`
- `ownership_map.md`

### 4. Architecture Thread
职责：

- 确定模块边界
- 确定技术栈和目录结构
- 明确环境初始化步骤
- 定义测试策略

交付物：

- `architecture.md`
- `setup_plan.md`
- `test_strategy.md`

### 5. Implementation Threads
职责：

- 在明确的文件边界内实现功能
- 提交局部变更
- 运行本线程负责的最小验证

规则：

- 不改不属于自己的模块
- 不跳过测试直接宣称完成
- 不解决需求层歧义

### 6. Review Thread
职责：

- 从缺陷、风险、回归角度审阅实现线程产物
- 检查改动和需求是否一致
- 检查安全、风格、边界条件

交付物：

- `review_report.md`
- `findings.json`

### 7. Test Thread
职责：

- 构建和补齐测试
- 跑单元、集成和端到端测试
- 记录失败证据
- 判断是否存在测试黑客行为

交付物：

- `test_report.md`
- `failing_cases.md`
- `verification_matrix.md`

### 8. Runtime Verification Thread
职责：

- 在真实运行环境中启动系统
- 验证关键用户路径
- 判断“能跑”与“符合设计”是否一致

交付物：

- `runtime_validation.md`
- `run_logs.md`
- `demo_checklist.md`

## 推荐状态机

### Phase 0: Intake
输入：设计文档、环境说明、仓库入口

通过条件：

- 能定位设计文档
- 能定位项目仓库或初始化目录
- 能确认最小运行环境

### Phase 1: Requirement Normalization
产物：需求包、假设清单、验收标准

闸门：

- 每条核心需求都可验证
- 不明确项被显式记录
- 非功能要求没有被遗漏

### Phase 2: Planning and Ownership
产物：任务图、线程分工、里程碑

闸门：

- 每个任务都有明确 owner
- 依赖关系可执行
- 没有两个线程同时拥有同一批文件的写权限

### Phase 3: Architecture and Setup
产物：目录结构、技术选型、环境脚本、测试策略

闸门：

- 环境能启动
- 最小 hello path 可运行
- 测试入口已经明确

### Phase 4: Parallel Implementation
产物：按模块拆分的实现结果

闸门：

- 每个线程提交本地验证结果
- 改动满足 ownership map
- 没有未解释的跨模块副作用

### Phase 5: Independent Review
产物：审阅报告

闸门：

- 高优先级问题为 0
- 中优先级问题已处理或记录风险接受理由

### Phase 6: Test and Repair Loop
产物：测试报告、失败用例、修复补丁

闸门：

- 单元测试通过
- 集成测试通过
- 关键端到端路径通过
- 没有明显 test hacking 证据

### Phase 7: Runtime Validation
产物：真实运行验证报告

闸门：

- 系统在目标环境中实际启动成功
- 关键用户路径可复现
- 运行日志与设计目标不冲突

### Phase 8: Delivery
产物：最终交付包

建议包含：

- 需求摘要
- 架构摘要
- 改动摘要
- 测试结果
- 真实运行结果
- 已知限制
- 下一步建议

## 并行执行规则

1. 并行线程必须拥有清晰的写入边界。
2. 公共文件需要显式锁或串行合并。
3. 审阅线程和测试线程不能与被审对象是同一线程。
4. 只要某线程发现需求歧义，必须回退到需求闸门，而不是私自补脑。
5. 线程状态要结构化记录，不能只留聊天记录。

## 测试分层

### 单元测试
验证局部逻辑正确。

### 集成测试
验证模块之间能连起来。

### 端到端测试
验证用户流程是否成立。

### 真实运行验证
验证项目在目标环境中真的能跑，而不是只在测试壳里通过。

## 失败处理策略

### 设计文档含糊
动作：停止扩写需求，输出假设清单和待确认点。

### 环境起不来
动作：进入环境故障排查，不继续伪造功能完成。

### 测试缺失
动作：先补最小复现测试，再进入实现。

### 多线程冲突
动作：冻结冲突文件，回到 ownership map 重新分配。

### 审阅发现高风险问题
动作：退回实现阶段，不允许直接带病进入验收。

## 必须记住的现实限制

- `DevEval` 类研究说明，完整软件生命周期自动化仍然不稳定。
- `TDFlow` 指出，真正的硬点常常是复现测试生成，而不是补丁本身。
- `OSWorld` 说明，真实电脑环境、多应用流程、GUI 操作仍是明显弱项。
- `GitGoodBench` 提醒，真正的软件开发不只是在文件里写代码，还包括 Git 工作流本身。

## 怎么用这份文档

最稳的用法不是把这里当“必须逐字照搬的圣经”，而是把它当成：

- 线程角色设计模板
- 执行闸门模板
- 风险控制模板
- 真实验证模板

如果要给 Codex 启动执行线程，优先结合 `papers.json` 和 `codex-prompt.md` 一起用。
